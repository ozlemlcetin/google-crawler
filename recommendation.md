# Recommendations

**Near-term improvements.** The most useful immediate change would be replacing the per-job JSON state files with SQLite. That gives atomic writes, fast lookups by `crawler_id`, and a cleaner path to resuming jobs without reading the whole file into memory. The crawler worker could also move from a single thread per job to a small pool using `ThreadPoolExecutor`, which would let each job fetch a few pages concurrently without much added complexity. On the deduplication side, a shared seen-set across all active jobs (rather than per-job `visited_urls` sets) would reduce the chance of the same URL entering multiple jobs' queues before either marks it visited.

**Longer-term.** Once this outgrows a single process, the URL queue could move to a message broker like Redis Streams so workers run as separate processes and scale independently of the search layer. Separating `CrawlerService` and `SearchService` also removes the GIL contention where a crawl-heavy workload competes with search requests for I/O. At that point it becomes worth adding structured logging (fields like `event`, `url`, `depth`, `duration_ms`) and tracking a few key metrics — query latency and queue depth are enough to catch most problems early. Distributing across multiple machines is a further step if the workload eventually justifies it.
